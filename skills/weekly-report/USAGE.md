# Weekly Report Skill 使用指南

## 快速开始（4 步）

### Step 1: 初始化季度（每季度一次）

```bash
bash skills/weekly-report/scripts/new-quarter.sh ~/weekly-reports
```

输出示例：
```
✅ 已创建 OKR 文件
---
OKR_FILE=/Users/you/weekly-reports/2026/Q1/okr.md
YEAR=2026 Q1

下一步: 编辑该文件，填写本季度 OKR
```

编辑 `okr.md`，填入本季度 OKR（参考 `reference/okr-template.md`），然后提交 Git。

### Step 2: 创建本周周报文件（每周一次）

```bash
bash skills/weekly-report/scripts/new-report.sh ~/weekly-reports
```

输出示例：
```
✅ 已创建周报
---
REPORT_FILE=/Users/you/weekly-reports/2026/Q1/03/03.23-03.29/report.md
WEEK=03.23-03.29
YEAR=2026 Q1 03月
OKR_FILE=/Users/you/weekly-reports/2026/Q1/okr.md
```

脚本自动完成：日期计算、目录创建、模板复制。

### Step 3: 粘贴流水账，调用 Skill

在 Claude 中说：

```
帮我写周报

这周主要做了：
- 完成了权限控制模块开发
- 修复了 3 个线上 bug
- 参加了产品需求评审会议

下周计划：
- 继续做权限控制的单元测试
- 优化发布系统性能
```

Skill 会自动：
1. 读取 `OKR_FILE`，获取 KR 和优先级
2. 解析你的流水账，提取任务和进度
3. 生成周报草稿，写入 `REPORT_FILE`
4. 列出所有`（待补充）`的字段

### Step 4: 填写缺失字段

Skill 输出后，你需要补充：

```markdown
## 上周工作总结

| 三只青蛙排序 | KR及其进展 | ... | 原期望的结果是什么 | ... |
|:---:|:---|:---|:---|:---|
| 🐸1 | 发布系统重构 | ... | （待补充） | ... |
```

改为：

```markdown
| 🐸1 | 发布系统重构 | ... | 完成权限控制模块，支持细粒度权限校验 | ... |
```

类似地填写：
- ✏️ **原期望的结果**（上周工作总结）
- ✏️ **期望的结果**（本周工作计划）
- ✏️ **核心措施**（本周工作计划，仅 ⭐⭐/⭐⭐⭐ 时）

### Step 5: 保存周报

```bash
# 保存到指定路径
# 路径格式：{year}/Q{quarter}/{month}/{week}/report.md
# 例如：2026/Q1/03/03.24-03.30/report.md

git add 2026/Q1/03/03.24-03.30/report.md
git commit -m "周报：2026-03-24 ~ 2026-03-30"
```

## 详细说明

### OKR 文件格式

OKR 文件使用 **Markdown 表格**格式，Skill 通过解析表格获取 KR 标题和优先级。

**关键点**：
- 每行一个 KR，列顺序固定
- KR 标题必须唯一（Skill 用标题匹配周报中的任务）
- 完整格式参考 `reference/okr-template.md`

**示例**：

```markdown
# Q1 2026 OKR

| 优先级 | KR 标题 | 目标 | 关键结果 | 负责人 | 开始日期 | 截止日期 | 状态 | 进度 | 备注 |
|:---:|:---|:---|:---|:---|:---:|:---:|:---:|:---:|:---|
| 🐸1 | 发布系统重构 | 完成发布系统 v2 上线 | 覆盖 100% 服务发布流程<br>发布时间降低 50% | wanghaidong | 01-06 | 03-31 | 🟡进行中 | 60% | 权限控制和通知模块已完成 |
| 🐸2 | 稳定性提升 | 降低 P0 故障率 | P0 故障率降低 50%<br>故障恢复时间 < 5 分钟 | team-stability | 01-06 | 03-31 | 🟡进行中 | 40% | 监控体系已搭建 |
```

### 周报字段说明

#### 必填字段（Skill 无法推断）

| 字段 | 位置 | 说明 | 示例 |
|:---|:---|:---|:---|
| **原期望的结果** | 上周工作总结 | 任务开始前的预期目标 | 完成权限控制模块，支持细粒度权限校验 |
| **期望的结果** | 本周工作计划 | 本周任务完成后的预期产出 | 完成权限控制模块的单元测试和集成测试 |
| **核心措施** | 本周工作计划 | 如何克服难点（仅 ⭐⭐/⭐⭐⭐ 时） | 1.复用已有框架<br>2.参考 team-stability 的测试用例 |

#### 自动填充字段（Skill 从 OKR 或流水账提取）

| 字段 | 来源 | 说明 |
|:---|:---|:---|
| 三只青蛙排序 | `okr.md` | 优先级（🐸1/🐸2/🐸3） |
| KR 名称 | `okr.md` | 关键结果标题 |
| 上周重要任务 | 流水账 | 你实际做的任务 |
| 实际产出/进展 | 流水账 | 完成情况、时间范围、工期 |
| 状态说明 | 流水账推断 | 🟢/🟡/🔴（你可以修改） |
| 本周重要任务 | 用户提供或推断 | 下周计划做什么 |
| 难度系数 | 推断 | ⭐/⭐⭐/⭐⭐⭐（你可以修改） |

## 常见场景

### 场景 1：某个 KR 这周没有进展

**流水账**：
```
这周主要做了 bug 修复和性能优化，没有做发布系统的工作
```

**Skill 的处理**：
- 在上周工作总结中，发布系统这一行的"上周重要任务"为空
- 在本周工作计划中，发布系统这一行的"本周重要任务"为空，标注`（待补充）`

**你需要做**：
- 确认是否需要在本周工作计划中为发布系统添加任务
- 如果没有，可以删除这一行

### 场景 2：流水账中提到的 KR 不在 OKR 中

**流水账**：
```
这周做了一个紧急的客户需求，涉及数据导出功能
```

**Skill 的处理**：
- 如果 `okr.md` 中没有"数据导出"这个 KR，Skill 会将其作为临时任务排在三只青蛙之后

**你需要做**：
- 确认是否需要将"数据导出"添加到 OKR 中
- 或者保持为临时任务即可

### 场景 3：某个任务跨越多周

**流水账**：
```
权限控制模块开发，从 3.18 开始，预计 3.31 完成，目前完成度 60%
```

**Skill 的处理**：
- 在"实际产出/进展"中记录：`【3.18～3.31】（10pd）（进行中）`
- 状态标记为 🟡（部分完成）

**你需要做**：
- 下周继续在本周工作计划中跟进这个任务
- 更新进度百分比

### 场景 4：某个任务完成度很高，但有风险

**流水账**：
```
权限控制模块开发完成，但在性能测试中发现瓶颈，需要优化
```

**Skill 的处理**：
- 完成度 > 80%，但有风险，标记为 🟡（而不是 🟢）

**你需要做**：
- 在"补充说明"中说明风险内容
- 在本周工作计划中添加"性能优化"任务

## 最佳实践

### 1. OKR 定义要清晰

❌ **不好**：
```markdown
## 🐸1 系统优化

**目标**：优化系统
```

✅ **好**：
```markdown
## 🐸1 发布系统重构

**目标**：完成发布系统 v2 上线，提升发布效率和稳定性

**关键结果**：
- 覆盖 100% 服务的发布流程
- 发布时间降低 50%
- 零故障上线率达到 99.9%
```

### 2. 流水账要具体

❌ **不好**：
```
这周做了很多工作，进展顺利
```

✅ **好**：
```
这周主要做了：
- 权限控制模块开发（3.18-3.22，4pd）
  - 服务权限控制
  - 权限校验中间件
  - 权限缓存优化
- 飞书通知集成（3.23-3.24，1pd）
  - 发布关键节点通知
  - 故障告警通知

进度：权限控制已完成，飞书通知在自测中
```

### 3. 期望结果要量化

❌ **不好**：
```
完成权限控制模块
```

✅ **好**：
```
完成权限控制模块，支持细粒度权限校验，覆盖 100% 服务，性能开销 < 5%
```

### 4. 核心措施要具体

❌ **不好**：
```
加强沟通，提高效率
```

✅ **好**：
```
1. 复用已有的权限校验框架，减少重复开发
2. 参考 team-stability 的单元测试用例，确保测试覆盖率 > 80%
3. 预留 2 天用于性能优化和压力测试
```

## 故障排除

### 问题 1：Skill 找不到 OKR 文件

**错误信息**（脚本实际输出）：
```
⚠️  首次使用？检测到季度 OKR 文件不存在：
    /path/to/reports/2026/Q1/okr.md

请先初始化该季度：
    bash skills/weekly-report/scripts/new-quarter.sh <reports-dir>
```

补录模式下会显示：`⚠️  补录模式：2026 年 Q1 的 OKR 文件不存在`

**解决方案**：
1. 运行 `new-quarter.sh` 初始化季度目录和 OKR 文件
2. 编辑生成的 `okr.md` 填写本季度 OKR
3. 重新运行 `new-report.sh`

### 问题 2：Skill 无法解析 OKR 文件

**可能原因**：表格格式不正确或列数不匹配。

**解决方案**：
1. 检查 `okr.md` 是否为标准 Markdown 表格格式
2. 检查表头列顺序是否正确（10 列）：优先级 | KR 标题 | 目标 | 关键结果 | 负责人 | 开始日期 | 截止日期 | 状态 | 进度 | 备注
3. 确保 KR 标题唯一，参考 `reference/okr-template.md`

### 问题 3：某个 KR 被标记为"未知 KR"

**错误信息**：
```
❓ 未知 KR: 数据导出
请确认是否需要添加到 OKR 中
```

**解决方案**：
1. 检查流水账中的 KR 名称是否与 `okr.md` 表格中的 KR 标题一致
2. 如果需要添加新的 KR，编辑 `okr.md` 表格，新增一行
3. 如果是临时任务，Skill 会自动将其排在三只青蛙之后，不关联 KR

## 文件结构参考

在你的 **weekly-reports 仓库**中，按以下结构组织：

```
weekly-reports/                  ← 你的周报数据仓库（独立仓库）
├── 2026/
│   ├── annual-review.md              ← 年度复盘（年末）
│   ├── Q1/
│   │   ├── okr.md                    ← 季度 OKR 定义（必须）
│   │   ├── quarterly-review.md       ← 季度复盘（可选）
│   │   ├── 01/
│   │   │   └── 01.05-01.11/
│   │   │       └── report.md         ← 周报
│   │   ├── 02/
│   │   └── 03/
│   │       ├── 03.17-03.23/
│   │       │   └── report.md
│   │       └── 03.24-03.30/
│   │           └── report.md
│   ├── Q2/
│   │   ├── okr.md
│   │   └── ...
│   ├── Q3/
│   │   ├── okr.md
│   │   └── ...
│   └── Q4/
│       ├── okr.md
│       └── ...
└── 2027/
    ├── annual-review.md              ← 下一年的年度复盘
    ├── Q1/
    │   ├── okr.md
    │   └── ...
    └── ...
```

**文件说明**：
- `annual-review.md`：年度复盘，包括全年 OKR 完成情况、亮点、挑战、经验教训、下一年展望等
- `okr.md`：季度 OKR 定义，每个季度一个
- `quarterly-review.md`：季度复盘（可选，暂无模板，用户自行编写）
- `report.md`：周报，每周一个

**注意**：
- `weekly-report` Skill 本身放在 `milian` 仓库中，周报数据放在 `weekly-reports` 仓库中，两个仓库是分离的。
- **跨年/跨月周**：当一周横跨两个月或两年时（如 12.28-01.03），脚本以**周一所在的年/月**作为归属目录。这意味着该周报会出现在 12 月目录下，即使 Sunday 已经是下一年。
