# Weekly Report Skill 使用指南

## 快速开始（3 步）

### Step 1: 创建季度 OKR 文件

```bash
# 复制模板到你的仓库
cp skills/weekly-report/reference/okr-template.md 2026/Q1/okr.md

# 编辑 okr.md，填入你的 OKR
# 例如：修改 🐸1/🐸2/🐸3 的标题、目标、关键结果等

# 提交 Git
git add 2026/Q1/okr.md
git commit -m "Q1 2026 OKR 定义"
```

**可选：创建年复盘文件**

```bash
# 年末时，复制年复盘模板
cp skills/weekly-report/reference/annual-review-template.md 2026/annual-review.md

# 编辑 annual-review.md，填入年度复盘内容
# 包括：OKR 完成情况、年度亮点、经验教训、2027 年展望等

# 提交 Git
git add 2026/annual-review.md
git commit -m "2026 年度复盘"
```

### Step 2: 粘贴流水账，调用 Skill

在 Claude 中说：

```
帮我写周报

这周主要做了：
- 完成了权限控制模块开发
- 修复了 3 个线上 bug
- 参加了产品需求评审会议

下周计划：
- 继续做权限控制的单元测试
- 优化发布系统性能
```

Skill 会自动：
1. 读取 `2026/Q1/okr.md`，获取 KR 和优先级
2. 解析你的流水账，提取任务和进度
3. 生成周报草稿
4. 列出所有【需你填写】的字段

### Step 3: 填写缺失字段

Skill 输出后，你需要补充：

```markdown
## 上周工作总结

| 三只青蛙排序 | KR及其进展 | ... | 原期望的结果是什么 | ... |
|:---:|:---|:---|:---|:---|
| 🐸1 | 发布系统重构 | ... | 【需你填写】 | ... |
```

改为：

```markdown
| 🐸1 | 发布系统重构 | ... | 完成权限控制模块，支持细粒度权限校验 | ... |
```

类似地填写：
- ✏️ **原期望的结果**（上周工作总结）
- ✏️ **期望的结果**（本周工作计划）
- ✏️ **核心措施**（本周工作计划，仅 ⭐⭐/⭐⭐⭐ 时）

### Step 4: 保存周报

```bash
# 保存到指定路径
# 路径格式：{year}/Q{quarter}/{month}/{week}/report.md
# 例如：2026/Q1/03/03.24-03.30/report.md

git add 2026/Q1/03/03.24-03.30/report.md
git commit -m "周报：2026-03-24 ~ 2026-03-30"
```

## 详细说明

### OKR 文件格式

OKR 文件使用 **Markdown + 结构化注释** 的格式，既易于人工维护，又易于 Skill 解析。

**关键点**：
- 每个 OKR 必须包含 `<!-- OKR_META ... -->` 注释块
- 注释块中的字段必须严格按照格式填写
- Markdown 部分可以自由调整，不影响 Skill 解析

**示例**：

```markdown
## 🐸1 发布系统重构

<!-- OKR_META
priority: 🐸1
title: 发布系统重构
objective: 完成发布系统 v2 上线，提升发布效率和稳定性
owner: wanghaidong
start_date: 2026-01-01
end_date: 2026-03-31
status: in_progress
progress: 60%
-->

**目标**：完成发布系统 v2 上线，提升发布效率和稳定性

**关键结果**：
- 覆盖 100% 服务的发布流程
- 发布时间降低 50%
- 零故障上线率达到 99.9%

**负责人**：wanghaidong  
**开始日期**：2026-01-01  
**截止日期**：2026-03-31  
**状态**：进行中  
**进度**：60%  
**备注**：当前完成了权限控制和通知模块，正在做性能优化
```

### 周报字段说明

#### 必填字段（Skill 无法推断）

| 字段 | 位置 | 说明 | 示例 |
|:---|:---|:---|:---|
| **原期望的结果** | 上周工作总结 | 任务开始前的预期目标 | 完成权限控制模块，支持细粒度权限校验 |
| **期望的结果** | 本周工作计划 | 本周任务完成后的预期产出 | 完成权限控制模块的单元测试和集成测试 |
| **核心措施** | 本周工作计划 | 如何克服难点（仅 ⭐⭐/⭐⭐⭐ 时） | 1.复用已有框架<br>2.参考 team-stability 的测试用例 |

#### 自动填充字段（Skill 从 OKR 或流水账提取）

| 字段 | 来源 | 说明 |
|:---|:---|:---|
| 三只青蛙排序 | `okr.md` | 优先级（🐸1/🐸2/🐸3） |
| KR 名称 | `okr.md` | 关键结果标题 |
| 上周重要任务 | 流水账 | 你实际做的任务 |
| 实际产出/进展 | 流水账 | 完成情况、时间范围、工期 |
| 状态说明 | 流水账推断 | 🟢/🟡/🔴（你可以修改） |
| 本周重要任务 | 用户提供或推断 | 下周计划做什么 |
| 难度系数 | 推断 | ⭐/⭐⭐/⭐⭐⭐（你可以修改） |

## 常见场景

### 场景 1：某个 KR 这周没有进展

**流水账**：
```
这周主要做了 bug 修复和性能优化，没有做发布系统的工作
```

**Skill 的处理**：
- 在上周工作总结中，发布系统这一行的"上周重要任务"为空
- 在本周工作计划中，发布系统这一行的"本周重要任务"为空，标注【待填写】

**你需要做**：
- 确认是否需要在本周工作计划中为发布系统添加任务
- 如果没有，可以删除这一行

### 场景 2：流水账中提到的 KR 不在 OKR 中

**流水账**：
```
这周做了一个紧急的客户需求，涉及数据导出功能
```

**Skill 的处理**：
- 如果 `okr.md` 中没有"数据导出"这个 KR，Skill 会标记为 `❓ 未知 KR`

**你需要做**：
- 确认是否需要将"数据导出"添加到 OKR 中
- 或者修改流水账，将其关联到现有的某个 KR

### 场景 3：某个任务跨越多周

**流水账**：
```
权限控制模块开发，从 3.18 开始，预计 3.31 完成，目前完成度 60%
```

**Skill 的处理**：
- 在"实际产出/进展"中记录：`【3.18～3.31】（10pd）（进行中）`
- 状态标记为 🟡（部分完成）

**你需要做**：
- 下周继续在本周工作计划中跟进这个任务
- 更新进度百分比

### 场景 4：某个任务完成度很高，但有风险

**流水账**：
```
权限控制模块开发完成，但在性能测试中发现瓶颈，需要优化
```

**Skill 的处理**：
- 完成度 > 80%，但有风险，标记为 🟡（而不是 🟢）

**你需要做**：
- 在"补充说明"中说明风险内容
- 在本周工作计划中添加"性能优化"任务

## 最佳实践

### 1. OKR 定义要清晰

❌ **不好**：
```markdown
## 🐸1 系统优化

**目标**：优化系统
```

✅ **好**：
```markdown
## 🐸1 发布系统重构

**目标**：完成发布系统 v2 上线，提升发布效率和稳定性

**关键结果**：
- 覆盖 100% 服务的发布流程
- 发布时间降低 50%
- 零故障上线率达到 99.9%
```

### 2. 流水账要具体

❌ **不好**：
```
这周做了很多工作，进展顺利
```

✅ **好**：
```
这周主要做了：
- 权限控制模块开发（3.18-3.22，4pd）
  - 服务权限控制
  - 权限校验中间件
  - 权限缓存优化
- 飞书通知集成（3.23-3.24，1pd）
  - 发布关键节点通知
  - 故障告警通知

进度：权限控制已完成，飞书通知在自测中
```

### 3. 期望结果要量化

❌ **不好**：
```
完成权限控制模块
```

✅ **好**：
```
完成权限控制模块，支持细粒度权限校验，覆盖 100% 服务，性能开销 < 5%
```

### 4. 核心措施要具体

❌ **不好**：
```
加强沟通，提高效率
```

✅ **好**：
```
1. 复用已有的权限校验框架，减少重复开发
2. 参考 team-stability 的单元测试用例，确保测试覆盖率 > 80%
3. 预留 2 天用于性能优化和压力测试
```

## 故障排除

### 问题 1：Skill 找不到 OKR 文件

**错误信息**：
```
❌ 找不到 OKR 文件: 2026/Q1/okr.md
请先创建该文件，参考模板: reference/okr-template.md
```

**解决方案**：
1. 确认你的仓库路径是否正确
2. 复制 `reference/okr-template.md` 到 `{year}/Q{quarter}/okr.md`
3. 编辑并提交 Git

### 问题 2：Skill 无法解析 OKR 文件

**错误信息**：
```
❌ OKR 文件格式错误，无法解析
请检查 <!-- OKR_META ... --> 注释块的格式
```

**解决方案**：
1. 检查每个 OKR 是否都有 `<!-- OKR_META ... -->` 注释块
2. 检查注释块中的字段是否完整（priority, title, objective, owner, status, progress）
3. 参考 `reference/okr-template.md` 的格式

### 问题 3：某个 KR 被标记为"未知 KR"

**错误信息**：
```
❓ 未知 KR: 数据导出
请确认是否需要添加到 OKR 中
```

**解决方案**：
1. 检查流水账中的 KR 名称是否与 `okr.md` 中的 `title` 完全一致
2. 如果需要添加新的 KR，编辑 `okr.md`，添加新的 OKR 块
3. 如果是临时任务，可以在周报中单独列出，不关联 KR

## 文件结构参考

在你的 **weekly-reports 仓库**中，按以下结构组织：

```
weekly-reports/                  ← 你的周报数据仓库（独立仓库）
├── 2026/
│   ├── annual-review.md              ← 年度复盘（年末）
│   ├── Q1/
│   │   ├── okr.md                    ← 季度 OKR 定义（必须）
│   │   ├── quarterly-review.md       ← 季度复盘（可选）
│   │   ├── 01/
│   │   │   └── 01.05-01.11/
│   │   │       └── report.md         ← 周报
│   │   ├── 02/
│   │   └── 03/
│   │       ├── 03.17-03.23/
│   │       │   └── report.md
│   │       └── 03.24-03.30/
│   │           └── report.md
│   ├── Q2/
│   │   ├── okr.md
│   │   └── ...
│   ├── Q3/
│   │   ├── okr.md
│   │   └── ...
│   └── Q4/
│       ├── okr.md
│       └── ...
└── 2027/
    ├── annual-review.md              ← 下一年的年度复盘
    ├── Q1/
    │   ├── okr.md
    │   └── ...
    └── ...
```

**文件说明**：
- `annual-review.md`：年度复盘，包括全年 OKR 完成情况、亮点、挑战、经验教训、下一年展望等
- `okr.md`：季度 OKR 定义，每个季度一个
- `quarterly-review.md`：季度复盘（可选），总结该季度的成果和经验
- `report.md`：周报，每周一个

**注意**：`weekly-report` Skill 本身放在 `milian` 仓库中，周报数据放在 `weekly-reports` 仓库中，两个仓库是分离的。
